# Residential Rental Market Analysis

## Етап 1: Збір даних (Web Scraping).
Під час розробки було досліджено офіційне API DOM.RIA, проте через жорсткі обмеження безкоштовного тарифу (rate limits), для масового збору історичних даних було прийнято архітектурне рішення написати власний веб-скрапер на Selenium.

Етап збору даних розпочався з детального аналізу HTML-структури та поведінки сайту DOM.RIA. Для забезпечення надійного та стабільного парсингу було розроблено скрипт на Python з використанням гібридного підходу:

1. *Двохетапна архітектура збору*: Написано дві ключові функції. `get_listing_urls` відповідає за швидкий збір посилань на сторінках пагінації (за допомогою **BeautifulSoup**), а `get_info` покроково обходить кожну квартиру і дістає детальні метрики.

2. *Обхід антибот-захисту*: Оскільки DOM.RIA має захист від автоматизації, для первинних запитів використано бібліотеку cloudscraper, а для взаємодії з динамічними елементами — **undetected_chromedriver**. Також впроваджено рандомізовані паузи (**random.uniform**) для імітації поведінки реального користувача.

3. *Стійкість до відсутності даних*: Створено власні функції-обгортки (`safe_find`, `safe_find_all`, `get_full_description`), які використовують блоки try-except. Завдяки цьому скрипт не "падає", якщо власник квартири не вказав якусь характеристику, а коректно записує значення "не знайдено".

4. *Безпечне збереження даних*: Головний цикл автоматизації загорнутий у блок перехоплення винятків (try...except KeyboardInterrupt...finally). Це гарантує, що навіть при примусовій зупинці роботи парсера або раптовому збої мережі, всі зібрані до цього моменту дані не будуть втрачені. Вони автоматично конвертуються в таблицю pandas і зберігаються у фінальний `.csv` файл.

## Етап 2: Очищення та збагачення даних (Data Cleaning & Feature Engineering).
Для налаштування логіки очищення було використано тестову вибірку зі 100 оголошень. Обробка проводилась за допомогою бібліотек pandas та регулярних виразів (re). Зібраний сирий текст було структуровано та перетворено на числові й категоріальні метрики для аналізу.

**Ключові трансформації:**

* *Парсинг числових значень та дат*: Очищення цін від спецсимволів, витягування площі та поверхів. Перетворення українських дат (наприклад, "12 бер") у стандартизований формат `datetime`.

* *Створення нових метрик (Feature Engineering)*: Написано кастомну функцію для перетворення текстового стажу рієлторів (напр., "1 рік 2 місяці") у загальну кількість місяців із подальшим ранжуванням на категорії (Новачок, Досвідчений, Професіонал, Експерт).

* *Вилучення прихованих ознак*: За допомогою регулярних виразів (re.search, .str.contains) із неструктурованих описів продавців витягнуто критично важливі для ринку бінарні маркери (1/0): готовність до блекаутів (газ, вода, інтернет без світла), наявність укриття та pet-friendly статус.

* *Агрегація даних*: Щоб не втратити інформацію, пошук характеристик (як-от укриття чи тварини) проводився одночасно по кількох колонках (теги, короткий та повний опис) із подальшим зведенням у єдиний достовірний стовпець.

**Результат:**  Формування чистого датасету (`cl_df`) розміром на 29 колонок із коректними типами даних (int64, float64, datetime64), який повністю готовий до розвідувального аналізу (EDA) та візуалізації.

## Етап 3: EDA (Exploratory Data Analysis). 
### 1. Обробка дублікатів та пропущених значень
* На першому етапі проведено глибоку фільтрацію бази від "рієлторського спаму" та технічних помилок. Видалено як повні дублікати (включно з повторами за `id` та `seller`), так і приховані копії оголошень з однаковими параметрами (адреса, поверх, кількість кімнат), де штучно змінювалися ціна або площа.
* Рядки з відсутніми критичними даними (поверх, адреса, місто) були видалені через їх незначну кількість. 
* Для відновлення пропущених значень житлової площі та кухні застосовано метод імпутації медіаною з групуванням за кількістю кімнат — це дозволило зберегти природний розподіл без спотворення загальної статистики.
* Пропуски в категоріальних даних (метро, район) замінено на логічні текстові заглушки, додатково згенеровано бінарну ознаку `has_metro`.

**У результаті очищення збережено 91% датасету (4550 із 5000 оголошень), що є високим показником якості після парсингу.**

### 2. Робота з викидами та логічні перевірки
Аналіз аномалій викидів у числових колонках базувався на візуалізації (Boxplots) та перевірці даних на відповідність реаліям ринку нерухомості.

* Екстремальні значення не видалялися сліпо: найдешевші об'єкти після ручної перевірки підтвердили свою достовірність (довгострокова оренда бюджетного сегмента), а багатокімнатні квартири та об'єкти з великою площею ідентифіковані як легітимні багаторівневі пентхауси (залишені в базі з рекомендацією використовувати медіану замість середнього для подальших розрахунків). 
* Для фільтрації преміум-сегмента створено метрику "ціна за квадратний метр", що дозволило точково знайти та видалити оголошення з хибними даними. 
* Також проведено сувору логічну фільтрацію: видалено квартири, де житлова площа або площа кухні перевищувала загальну, відфільтровано неможливі комбінації поверхів та обмежено максимальну кількість кімнат до 6. 
* Бінарні та категоріальні змінні успішно пройшли перевірку на коректність, а колонки-ідентифікатори (ідентифікатори, посилання, дати) були виключені з пошуку аномалій.
